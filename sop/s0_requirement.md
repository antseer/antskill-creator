# S0 — 需求采集 + Demo 粗版

> **v5 重组**:合并需求采集和粗 demo,让用户尽早看到"东西",对齐预期。

## 阶段目标

1. 产出一份填满的**需求画布**(7 题),作为后续所有阶段的对齐基准
2. 产出一份**粗版 Demo**(demo-v0.html),让用户所见即所对齐

## 入口条件

- 用户说"做/建/创建一个 XXX Skill"或"帮我做一个 skill / skill PRD"
- 或带着零碎想法进来(此时先做需求结晶再填画布)

## 方法论

- **先理解卡后画布后 demo**。理解卡(对人)→ 画布(对工厂)→ Demo(让用户点头),三步缺一不可。
- **理解卡用主张提问,画布用追问填空**。两者方法不同,不要混用。
- **Demo 是"粗版",不是 Hi-Fi**。目标是让用户看了点头说"对,就这个感觉",而不是最终交付品。Hi-Fi 在 S3 做。

详见 `methodology/intent-anchoring.md`。

## Part 0 — 需求理解卡(必做,在画布之前) ⭐ v0.3 新增

参考 `templates/requirement/intent-card.md`。

### 核心原则
- 用主张提问,不要开放式提问("你想要 A 还是 B,我倾向 A 因为 X")
- 复述用户原话,不要改写
- 2-4 个关键决策点,不多不少
- 三个出口:确认 / 修改某项 / 完全重来
- 1 屏可读完,< 30 行

### 6 问框架(挑模糊的问,明确的不问)

| 层 | 问题 |
|---|---|
| 意图层 | ① 用户想解决的本质问题是什么(不是字面问题) |
| 边界层 | ② 一次性需求 / 可复用工具? ③ 范围固定 / 用户输入? |
| 立场层 | ④ skill 多大程度替用户做判断?事实 / 指引 / 建议 |
| 形态层 | ⑤ 输出是简短摘要 / 完整答卷 / 交互式探索? |
| 诚实层 | ⑥ 用户能接受的"承认不确定"程度?藏着 / 显眼 / 拒绝输出 |

### 触发逻辑

| 情境 | 阶段 0 行为 |
|---|---|
| 用户首次调用 skill | 必触发,完整理解卡 |
| 用户调用过、参数明确 | 极简版:1 行复述 + [继续/调整] |
| 用户在前序对话讨论过 | 跳过,但在答卷顶部一行注明"按你之前说的 X 方式生成" |

### 产出物

| 文件 | 内容 |
|---|---|
| `intent-card.md` | 理解卡 markdown,作为 S0 第一份产物 |
| `intent-card-confirmed-by-user` | 用户确认时间戳(可以是对话片段引用) |

理解卡确认后,才进 Part A 画布。

## Part A — 需求画布(7 题必填)

### Q1. WHO — 用户是谁?
- 用户画像(经验水平、知识背景、决策习惯)
- 用户痛点(现在怎么做的?哪里不爽?)
- 典型使用场景(什么时候/在哪打开这个 Skill?)

### Q2. WHAT — 做什么?
- **核心功能**:一句话描述(能被用户复述出来)
- **关键输入**:用户要提供什么?
- **关键输出**:用户期望看到什么?(结论 / 数据 / 建议 / 预警)

### Q3. WHY — 为什么做?
- 现有替代方案是什么?
- 比替代方案好在哪?(更快 / 更准 / 更全 / 更省事)
- 不做的后果是什么?

### Q4. WHERE — 数据从哪来?
- **主数据源**:具体到 API / MCP Tool / 爬虫 / 手动输入
- **辅助数据源**
- **数据刷新频率要求**(实时 / 分钟 / 小时 / 天)

### Q5. WHEN — 时效性要求
- 实时 / 准实时(< 1min)/ 定期(小时 / 天)?
- 用户可接受的最大延迟?

### Q6. HOW — 交互形式
- 一次性结果 vs 持续监控?
- 主动推送 vs 被动查询?
- 移动端 / 桌面端 / 两者?

### Q7. PARADIGM — 范式选择 ⭐
- (a) Skill 本身就是最终产品,直接运行 → **A 实现型**
- (b) 需要产出规范文档,指导后端开发 → **B 规范型** ⭐默认
- (c) 两者都要 → **C 双模**

## 半成品边界沟通(Q7 范式确认后必做)

在 Q7 确定范式后,向用户**明确沟通**工厂交付的边界:

> 我们这个工厂产出的是"分层半成品":你会拿到**完整的 PRD + 高保真前端 + 按 L1-L5 分层组织的 skill 工程目录**,
> 其中 **L1-A(MCP 现有)** 部分可以直接调,**L3 脚本 + L4 prompt** 有可运行版本或壳子+TODO,
> 但 **L1-B(MCP 新建原始工具) + L2(MCP 新建聚合接口)** 需要后端同事按我们给出的接口契约补齐。
> 这样可以吗?如果你需要的是"立刻能跑的东西",请选范式 A。

## Part B — Demo 粗版(demo-v0.html)

画布确认后,开始出粗版 demo。

### 原则

- **粗版 ≠ 丑版**。布局、信息层级、组件类型要对;视觉细节(最终色、字体、动效)不用追求。
- **覆盖 P0 清单每一项**。画布里 Q2 的关键输出每一条都要在 demo 上有对应组件。
- **合理假数据**。用程序生成或手写一份贴近真实的 mock(不是 lorem ipsum,不是 NaN/undefined)。
- **状态完整**。默认态 / 有数据态 / 空态至少各有一屏(loading/error 可以先不做,S3 再补)。

### 约束

- 单个 HTML 文件(CSS + JS 内联)
- **不允许使用 localStorage / sessionStorage**(用内存状态)
- 底部标注 `demo-v0 · Mock Data · Powered by Antseer.ai`

### 金字塔原则(粗版也要遵守)

- **Hero 区必须有结论**(不允许只放输入表单)
- 中间层放支撑结论的 1-3 个可视化
- 底部放信任层(方法/风险/原始数据,可折叠)

### 交付给用户看

demo-v0 做完后,向用户逐屏展示,让用户:
1. 确认"对,就这个感觉" → 进 S1
2. 或提出调整 → 在 S0 内迭代,不跳步

## 执行步骤总览

```
0. 【v0.3 新增】产出需求理解卡 intent-card.md(主张+决策点)
0.5 用户确认理解卡(✓ 继续 / ✎ 修改 / ✗ 重来)
1. 展示 7 题画布模板
2. 收集用户回答(基于已确认的理解卡展开)
3. 逐题检查是否有实质性回答(非"待定"/"随便"/"都行")
4. 对模糊回答启动追问
5. 画布填满后,复述让用户确认
6. 讲清半成品边界,确认范式
7. 查 design-system 了解 Antseer 基础气质
8. 出 demo-v0.html 粗版(屏 -1 必须实际渲染理解卡)
9. 向用户展示 demo-v0,确认"就这个感觉"
10. 执行 G0 门禁
```

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 追问 3 轮后 Q4 数据源仍模糊 | 停止,让用户先去弄清数据源再回来 |
| 用户坚持"都可以 / 随便" | 提供 2 个选项让 ta 二选一 |
| 需求超出工厂边界(如要做纯交易系统) | 告知边界,引导到产品需求或外部团队 |
| demo-v0 用户说"不对" | 在 S0 内继续迭代,不允许进 S1 |
