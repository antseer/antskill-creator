# Visual Registry — 已建 Skill 视觉登记表

> **每个新 Skill 在 S2 开始前必须查此表**，确保**主色 × Display 字体 × Hero 类型**三维组合不与已有 Skill 重复。
> S2 完成后必须把新 Skill 登记到本表，否则 G2 FAIL。

---

## §1 为什么要差异化？

Antseer 的基础色板与布局是强统一的（保持家族一致性）。
但每个 Skill 需要有自己的**记忆点**，否则在 Skill 商店或 Skill Hub 里相互之间难以区分。

差异化体现在三个维度：

| 维度 | 可选池 | 作用 |
|------|--------|------|
| **主色** | `color-pool.md` | 决定这个 Skill 的"气质"（警戒色 / 分析色 / 交易色 / 社区色）|
| **Display 字体** | `display-font-pool.md` | 决定 Hero 区的"人设"（编辑部 / 学术 / 控制台 / 报纸）|
| **Hero 类型** | 见 §3 | 决定"结论怎么呈现"（榜单卡 / 仪表盘 / 时间轴 / 对比矩阵）|

| Hero 类型 | 描述 | 适用场景 | 已占用 |

**新 Skill 必须选一个"未占用"的 Hero 类型**。如果 8 种都用完了，则需要先提案扩展此池。

## §4 登记流程（S2 开始前 / S2 结束后）

### S2 开始前：选值

```
1. 读 color-pool.md → 挑一个未被使用的主色
2. 读 display-font-pool.md → 挑一个未被使用的 Display 字体
3. 查本表 §3 → 挑一个未被使用的 Hero 类型
4. 暂存选择，进入 S2 设计 Demo
```

### S2 结束后：登记

在本表 §2 追加一行：

```
| {Skill Name} | {slug} | #xxxxxx | {色名} | {字体} | {Hero 类型} | {气质} | {YYYY-Qn} |
```

### 冲突处理

| 冲突场景 | 处理 |
|---------|------|
| 主色被占用 | 从 color-pool 再选一个 |
| Display 字体被占用 | 从 display-font-pool 再选一个 |
| Hero 类型被占用 | 选一个未占用的，或者用同类型但**显著不同的视觉重心**（例：都是"大卡+统计小卡"，但一个统计卡在右侧一个在下方）|
| 三维组合都找不到 | 提案扩展色池 / 字体池 / Hero 类型池 |

---

## §5 扩展池的申请

如果 `color-pool.md` / `display-font-pool.md` / Hero 类型池里可选项即将耗尽（剩 ≤ 2 项），应向工厂维护者提案扩展，不能在造 Skill 时临时加新色。

**临时加色**会破坏：
- Skill Hub 的家族一致性
- 已登记 Skill 之间的区分度计算
- Claude 下次做 Skill 时的选择空间推理

---

## §6 本登记表的 SoT 地位

- 本表是**视觉差异化的唯一裁决源**
- G2 门禁会读取本表验证新 Skill 的三维组合
- 本表与单个 Skill 里的 `frontend-notes.md` / `viz-specs.md` 冲突时，以本表为准
- 本表只记录"已建 Skill"，不记录"拟建"或"草案"
