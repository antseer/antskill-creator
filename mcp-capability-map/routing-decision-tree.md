# L1-A / L1-B / L2 / L3 / L4 / L5 归属判定决策树(完整版)

> 本文件是 S2 阶段判定每个数据点归属的**权威决策树**。比 SOP 里的简版更详细,涵盖边界情况。

---

## §1 六层定义(复习)

| 层 | 本质 | 触发特征 | 责任方 |
|---|---|---|---|
| **L1-A** MCP 现有 | MCP 已有工具直供 | 本轮同步的真源文档里找得到 | Skill 运行时直调 |
| **L1-B** MCP 缺失 | MCP 原始数据缺,需新建 | 原始数据类,真源里没有 | 后端(写进 data-prd.md) |
| **L2** MCP 新聚合 | 跨实体/时间/维度聚合,多 Skill 复用 | 月度 GMV、用户画像、7 日均线 | 后端(写进 data-prd.md) |
| **L3** Skill 脚本 | 本 Skill 私有、轻量、毫秒级计算 | Bollinger、归一化、Z-Score | Skill 开发者(PM 产出) |
| **L4** LLM 结构化 | 自然语言或主观判断输出 | 摘要、建议、风险叙事 | Skill 开发者(PM 产 prompt+Fallback) |
| **L5** 前端展示 | HTML/CSS/JS 组件渲染 | 卡片、图表、表格、弹窗 | Skill 开发者(PM 出 HTML) |

辅助标记:`INPUT`(用户输入)、`STATIC`(静态文案)——不走层级判定但要记录。

## §2 完整决策树

```
从 S1 数据盘点清单中取一条数据点 D
│
├── D.性质 == "用户输入" ?
│   └── YES → INPUT(不走层级判定)
│
├── D.性质 == "静态文案" ?
│   └── YES → STATIC(写入 i18n/copy 或直接写死在 HTML)
│
├── D.性质 == "自然语言" ?
│   └── YES → L4
│              ├─ 必须有结构化输入 Schema(来自 L1/L2/L3 的结果)
│              ├─ 必须有结构化输出 Schema
│              └─ **必须有 Fallback 模板函数**
│
└── D.性质 == "原始值" / "衍生值" / "时序数据" / "关联数据" / "布尔判定" ?
    │
    ├── Q1: 这个数据是"原始值(不需要计算)"吗?
    │   ├── YES
    │   │    └─ 查本轮同步的 MCP 真源缓存
    │   │       ├── 找到对应 Tool → **L1-A**(MCP 已有)
    │   │       └── 找不到         → **L1-B**(MCP 需新增原始工具,附期望接口形态)
    │   │
    │   └── NO(需要计算)→ 进入 Q2
    │
    ├── Q2: 这个数据需要"跨实体/时间/维度的大量聚合"吗?
    │       (例:月度 GMV、7 日均线、全平台 TOP 50、用户画像)
    │   ├── YES
    │   │    └─ **L2**(MCP 应新增聚合接口,附期望接口形态 + 降级路径)
    │   │       原因: 不应让 Skill 脚本每次请求都跑大量聚合;
    │   │             多 Skill 会复用同一聚合;
    │   │             聚合结果有时效性需缓存。
    │   │       例: "过去 30 天全平台稳定币收益 TOP 50" / "月度 GMV 序列"
    │   │
    │   └── NO(轻量、本 Skill 特有)→ 进入 Q3
    │
    ├── Q3: 这个数据是"主观判断 / 自然语言"吗?
    │       (有些字段看起来是衍生,实际需要 LLM 解读)
    │   ├── YES → **L4**(补 prompt + Fallback)
    │   └── NO  → 进入 Q4
    │
    └── Q4: 纯展示层的视觉规则?
        (根据某数值改变颜色/大小/图标的纯映射规则)
        ├── 规则极简单(直接 CSS 类映射)→ **L5**(前端 JS 根据值渲染)
        └── 规则复杂(多阈值/多维度组合)→ **L3**(产出分类字段,L5 只做映射)
```

## §3 边界情况与决策指引

### B1 · 某字段是"当前值",而时序是 L1-A,归哪?

例:D10 当前概率,依赖 D30 概率时序。

- 若 `ant_polymarket` 有 `get_current_price` 这种单点工具 → D10 归 **L1-A**(独立调用)
- 若只有 `get_price_history` 时序工具 → D10 从 D30 取 `[-1]`,归 **L3**(Skill 自己取尾点)
- 不要混成 L1-A,因为 L1-A 必须有独立的 MCP 调用路径

### B2 · 某字段依赖多个 L1-A + 一点计算,归哪?

例:D11 = D10 当前 - D12 24h 前,两个都是 L1-A。

- 计算本身轻量 → D11 归 **L3**,依赖 D10 + D12
- 若计算复杂(多源 + 时间对齐 + 大量窗口)→ 归 **L2**,新建 `ant_polymarket.price_delta(market_id, window)`

### B3 · 跨 Skill 复用但目前只有本 Skill 用,归哪?

例:D70 月度 GMV,当前只有本 Skill 用,但很可能其他 Skill 未来也用。

- **归 L2**,登记 data-prd.md,优先级 P1
- 在 data-prd.md 的降级路径里写"暂时可用 L3 脚本 `gmv.py` 基于 price_history 求和,性能差但可用"
- 后端实现后,L3 脚本加"优先调 MCP,回退本地"的开关

### B4 · LLM 生成一个"类别"(如"观望/加仓/减仓"),算 L4 还是 L3?

- 判断依据:**是否需要理解上下文语义**?
  - 需要(如综合多个指标 + 新闻 + 历史案例给出判断)→ **L4**
  - 不需要(如简单阈值 `if z > 2: 高;  else: 低`)→ **L3** 产出 `risk_level`,L5 映射文字

### B5 · 显示"某字段在哪个区间(高/中/低)"

- 阈值固定 → **L3** 产出 `level`,L5 映射颜色/文字
- 阈值需 LLM 根据上下文调整 → **L4**
- 阈值是全局配置,无需计算 → **L5** 直接硬编码阈值做映射

### B6 · Tooltip 里显示"公式是 X"

- 公式本身是静态文案 → **STATIC**
- 公式里代入当前值(如"MA = 62%")→ STATIC 模板 + **L3** 填充变量

### B7 · Hero 结论是"建议观望",归哪?

- 这是自然语言 + 主观判断 → **L4**
- 即使背后规则可能是 L3 的 risk_level,呈现为"建议观望,因为..."这类文案属于 L4
- L4 的输入 schema 应包含 L3 的 `risk_level` + 其他上下文

### B8 · 某数据点既像 L1-B 又像 L2,怎么区分?

- L1-B 是**原始数据**,MCP 一条查询返回一批(例:新闻列表、订单列表)
- L2 是**聚合结果**,MCP 一次查询返回计算后的结论(例:月度 GMV、Top 50)
- 看 MCP 侧的实现成本:读一张表就能返回 = L1-B;需要跨表/跨时间汇总 = L2

## §5 常见判定错误

| 错误 | 正确归属 | 理由 |
|---|---|---|
| 把"月度 GMV"放 L3 | **L2** | 跨时间聚合 + 多 Skill 复用 + 应 MCP 缓存 |
| 把"Top 50 稳定币收益"放 L3 | **L2** | 跨平台聚合量大,不应 Skill 实时跑 |
| 把"7 日均线"放 L3 | **L2**(若跨时间大量计算) / **L3**(若窗口小、Skill 特有) | 看窗口大小和复用度 |
| 把"风险等级分类(规则简单)"放 L4 | **L3** | 规则清晰无需 LLM,归 L3 省成本 |
| 把"推荐理由"放 L3 | **L4** | 自然语言输出必属 L4 |
| 把"从时序取末尾点"放 L1-A | **L3** | 这是 Skill 自己在本地取,MCP 不知道这个字段存在 |
| 把"Bollinger 上轨"放 L5 | **L3** | 计算不应在展示层,L5 只渲染 |
