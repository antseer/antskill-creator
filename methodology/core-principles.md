# 底层方法论 — 核心原则

## 1. 金字塔原则（Pyramid Principle）

所有 Skill 的信息架构必须遵循金字塔结构：

```
          ┌─────────────┐
          │   结论/推荐   │  ← Hero 区：3 秒内理解
          ├─────────────┤
          │  论据/可视化  │  ← 中间层：支撑结论的数据
          ├─────────────┤
          │ 原始数据/方法 │  ← 信任层：可折叠
          └─────────────┘
```

**反模式**：先展示一堆数据，最后才给结论。这是「倒金字塔」，用户看到一半就走了。

**正确做法**：Hero 区直接说「推荐 A，因为 X」，然后用图表证明 X，最后把原始数据放在可折叠区域。

## 2. 价值感 → 信任感管线

用户的心理路径是：

```
看到结论 → "这对我有用吗？"（价值感）
  → 看到论据 → "凭什么这么说？"（信任感）
    → 看到方法论 → "我信了"（决策）
```

**每一层缺失都会中断管线。** 只有结论没有论据 = 空口白话；只有数据没有结论 = 信息垃圾。

## 3. 先穷举后归属

**在判定"谁来算"之前,必须先穷举"要什么"**。

```
❌ 错误顺序: 先审 MCP 能力 → 再设计页面
  结果: 页面加新数据就要重审,MCP 审计白做

✅ 正确顺序:
  S1 先把页面每个元素穷举到底(数据盘点清单,来源全"待定")
  S2 再逐条判定每个数据的归属(L1-A / L1-B / L2 / L3 / L4 / L5)
```

对应的 S1 和 S2 顺序在 v5 中不允许颠倒。

## 4. 五层职责分离原则

当前流程采用单轴 **L1-A / L1-B / L2 / L3 / L4 / L5** 六分法。

| 层 | 只负责 | 约束 |
|---|---|---|
| **L1-A** MCP 现有 | 调用 MCP 已有工具 | 不做计算、过滤、排序。只做调用 → schema 映射 |
| **L1-B** MCP 缺失(新建原始) | (由后端实现) | 工厂只写期望接口形态,不自己兜底 |
| **L2** MCP 新聚合 | (由后端实现) | 跨实体/时间/维度聚合,多 Skill 可复用 |
| **L3** Skill 脚本 | 本 Skill 私有的轻量纯函数计算 | 纯函数,相同输入产相同输出;不调 API 不调 LLM |
| **L4** LLM 结构化 | 自然语言 / 主观判断 | **必须有 Fallback**;不做数学计算 |
| **L5** 前端渲染 | 接收上游数据,灌进组件 | 不做计算不做判断,纯渲染(简单 UI 映射除外) |

**为什么分离?**
- L1-A 出问题 → 只需要修 API 调用,不影响算法
- L1-B / L2 未实现 → 登记 data-prd.md 交后端,不污染 L3
- L3 出问题 → 单测快速定位
- L4 出问题 → 调 Prompt + Fallback 就行,不碰代码
- L5 出问题 → 改 CSS/组件,不碰逻辑

## 5. 归属单一原则

一个数据点只归一层,不跨层持有。若看起来需要跨层,说明该拆成两个 Dxx:
- 例:"当前价" 既是 L1-A 的原始值,又是 L3 "从时序取末尾"的派生 → 拆成 D30(L1-A 时序)+ D10(L3 从 D30 取尾)
- 例:"推荐理由" 基础分是 L3 算的,但文案是 L4 生成的 → 拆成 Dxx-score(L3)+ Dxx-reason(L4)

PRD 的每个模块按 L1-L5 组织,不涉及的层显式声明"本模块不涉及 L?,原因..."。

## 6. 降级优雅原则

每一层都必须有降级方案：

| 层 | 正常 | 降级 |
|----|------|------|
| L1-A | MCP 工具直调 | Fallback 数据源 → 缓存 → 空数据 + 错误卡 |
| L1-B | MCP 新建原始工具 | 未实现时:前端隐藏该组件,data-prd.md 记录用户感知 |
| L2 | MCP 新建聚合接口 | 未实现时默认登记到 data-prd.md；仅当 L1 原始数据已真实覆盖、L3 只是临时性能降级、且 data-prd.md 仍保留 L2 缺口时，才允许 L3 临时实现 |
| L3 | Skill 脚本完整计算 | 跳过可选字段 → 用默认值 → 标注"数据不完整" |
| L4 | LLM 正常调用 | **Fallback 模板函数**(强制必填)→ 结构化输出保持一致 |
| L5 | 完整渲染 | 隐藏缺失组件 → 显示骨架屏 → 全局错误卡 |

## 7. 风险对称原则

**有推荐就必须有风险提示。** 推荐和风险必须在视觉上等权重呈现。

反模式：大字推荐 + 小字风险 → 误导用户。
正确做法：推荐卡下方紧跟同等视觉权重的风险卡。

## 8. 异常叙事化原则

数据异常不能只打标签（"⚠ 异常"），必须翻译成人话：

❌ `APY_OUTLIER: true`
✅ "这个产品的年化收益率 (28.5%) 是同类的 4.7 倍，其中 82% 来自代币激励。如果代币价格下跌 50%，实际收益率将降至约 6.1%。"

## 9. 数据来源透明原则

每个数据点必须可追溯：
- 数据来源（哪个 MCP Tool / 哪个协议）
- 更新时间（UTC 时间戳）
- 计算方法（如果是衍生指标，说明公式）
- 层级归属（L1-A / L1-B / L2 / L3 / L4 / L5）

## 10. 半成品自知之明

工厂不产出生产系统，只产出**施工图纸**。任何时候工厂试图"模拟真实 API 调用"或"写一个能跑的生产脚本"时，必须停下来问自己：这是不是越权？

边界详见 `semi-finished-boundary.md`。

## 11. 不做交易代理

Skill 的边界是**决策辅助**，不是交易执行。

- ✅ 可以：推荐产品、对比方案、模拟收益、风险分析
- ❌ 不可以：一键下单、自动转账、代签合约
