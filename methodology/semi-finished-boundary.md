# 半成品边界声明（v5 保留 v4 核心）

> Skill 工厂是"图纸工厂"，不是"生产工厂"。本文件定义工厂产出物的边界。

---

## §1 为什么工厂只产半成品？

**根本原因：生产系统的关注点不在工厂的能力范围内。**

生产系统需要：
- 真实的 MCP Server 实现（需要后端工程师做数据采集、入库、API 设计）
- 认证、限流、监控、告警（需要 SRE/平台工程）
- CI/CD、灰度、回滚（需要 DevOps）
- 长期维护、版本演进、性能优化（需要工程团队）

**工厂唯一擅长的事**：把"用户想要什么"翻译成"工程师能动手做什么"。这就是半成品的价值。

## §3 范式 A/B/C 下的边界差异

### B 范式(默认):纯半成品

- 交付分层 Skill 工程 + 高保真前端 + 数据 PRD
- L3/L4 **以 PM 可产出为准**:L3 可运行或壳子+TODO,L4 prompt + Fallback 必有
- L1-B / L2 不产出实现,仅产出**期望接口形态**交后端
- 后端工程师参照 `data-prd.md` 和 `layers/` 在产品代码库实现

### A 范式:半成品 + 原型代码

- 额外要求 L3 脚本必须可运行 + 单测全过
- L4 必须有真实 LLM 调用的可运行版本(+ Fallback)
- L1-B / L2 可用本地 stub 代替 MCP,但要在 `review-report.md` 明确标注
- **原型代码仅供验证算法,不是生产代码**

### C 范式:A + B

两边都做,交付清单双份。

## §4 半成品声明(S5 强制)

v5 中半成品声明分布在三处,而非单一 `HANDOFF-REVIEW.md`:

1. **交付版 SKILL.md 的 frontmatter**:
   - `status: semi-finished`
   - `layers` 字段列出 L1-A/L1-B/L2/L3/L4/L5 的具体条目
   - `mcp_gap_summary` 汇总 P0/P1/P2 数量

2. **data-prd.md**:每个 L1-B / L2 缺口必含
   - 对应数据点 Dxx
   - 调用场景
   - **期望接口形态**(endpoint / 入参 / 出参 / 刷新频率)
   - 降级策略(若后端未实现,Skill 如何表现)
   - 优先级 P0/P1/P2

3. **review-report.md**:S4 产出,记录 HTML ↔ 双 PRD 的对齐情况和 v2 延期项

## §5 边界冲突的裁决

当用户要求工厂做"越界"工作时(例:"帮我把这个 Skill 真正跑起来"):

1. **明确拒绝生产级工作**,但可以做原型级验证(例:用 Mock 数据跑一次完整 Pipeline)
2. **告知用户边界**:这部分需要后端工程师介入
3. **把要求转化为缺口**:写入 `data-prd.md`,标清楚优先级 P0/P1/P2

**话术模板**:

> "这个功能属于生产系统范畴(真实 MCP 调用 / 认证 / 缓存),不在工厂交付范围内。
> 我已经把它作为 P0 优先级写入 `data-prd.md`,后端工程师参照期望接口形态实现即可。
> 如果你需要原型级验证,我可以用 Mock 数据跑一次 L1→L2→L3 流程(A 范式)。"

## §6 质量门禁对边界的强约束

- **G2** 强制要求:所有 L1-B / L2 条目必附期望接口形态(endpoint / 入参 / 出参 / 刷新频率 / 降级策略)
- **G2** 强制要求:所有 L4 条目必有 Fallback
- **G5** 强制要求:交付版 SKILL.md frontmatter 含 `status: semi-finished` + `layers` + `mcp_gap_summary`
- **G5** 强制要求:data-prd.md 按 P0 / P1 / P2 分级
- **G5** 强制要求:端到端可追溯(L5 → L4/L3 → L2 → L1 无断点,断点必须是已登记的 gap)

任何阶段的 Claude 试图"帮用户把生产系统做完",都会在对应 Gate 上被打回。
