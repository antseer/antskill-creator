# 需求锚定澄清(Intent Anchoring)— 方法论

> **新增于 skill 包 v0.3。所有 Skill 在 S0 进入"7 题画布"之前,必须先做一次"需求锚定澄清",产出一张"理解卡"给用户校验。**

## §1 为什么需要这一步

用户提出的初始需求 ≠ 用户真正想要的产品价值。中间有一段"意图理解 + 价值锚定"的距离。

如果跳过这一步直接进 7 题画布,会发生:

- 画布填得很快,但底层假设是错的(画布是基于"我以为的需求"而非"用户真正的需求"填的)
- 进 demo-v0 后用户说"不是这个意思",倒推回去发现是 S0 一开始就理解错了
- 多轮返工,而每一轮都在错误的方向上越走越远

**痛点案例**(真实发生过):
- 用户说"做个财报对加密的传导分析",我直接画了 3 个长得差不多的 demo,因为我没问"差异化重要吗"
- 用户说"看 K 线",我堆了一堆文字,因为我没确认"图先于字还是字先于图"
- 用户说"三类财报",我塞进同一个统计模型,因为我没问"三类的传导机制是不是同一回事"

每一次返工的根源都是同一个:**用户提出的需求和我以为的需求之间有 gap,但我没主动去消除这个 gap**。

## §2 核心原则

### 2.1 带主张地问,不要开放式问

❌ 开放式:"你想要什么?"  
✅ 带主张:"你想要 A 还是 B?我倾向 A,理由是 X。"

带主张的好处:
- 用户决策成本低 1 秒拍板
- 暴露了你的隐含假设,用户能直接纠偏
- 即使用户全都同意你的倾向,也是一次主动确认而非被动跳过

### 2.2 复述用户原话,而不是改写

理解卡的"问题复述"必须是用户原话或最低限度改写,不要"翻译"。  
如果用户说"做个财报传导分析",不要写成"构建一个跨市场宏观影响因子量化模型"。  
让用户一眼看到的是他自己说过的话,他才会相信你听进去了。

### 2.3 输出主张 + 理由,不输出待办事项

❌ 不要写:"接下来我要去查 color-pool / 选 Hero 类型 / 画原型..."  
✅ 要写:"我打算给你的是 X(一句话定义),理由是 Y。"

用户不在意你怎么干活,在意你打算交付什么。

### 2.4 三个出口,不要追问

理解卡确认时只给三个按钮:
- ✓ **确认,继续生成**
- ✎ **修改某项**(进入对话模式针对那一项澄清)
- ✗ **完全重来**(我理解错了)

不要在卡片里嵌套"还有 5 个问题等你回答",那等于又回到开放式提问。

### 2.5 不是每次都全量跑

| 情境 | 阶段 0 行为 |
|---|---|
| 用户首次调用 skill | 必触发,完整理解卡 |
| 用户调用过、参数明确(比如指定 "COIN→BTC") | **极简版**:1 行复述 + "[继续 / 调整]" |
| 用户在前序对话里已经讨论过细节 | **跳过**,直接进 demo,但在答卷顶部一行注明"按你之前说的 X 方式生成" |

如果阶段 0 比阶段 1 还长,就本末倒置了。

## §3 6 问框架

理解卡背后是一套 6 问,每次只挑 2-4 个不明确的问,明确的不问。

```
意图层:
  问题 ① 用户想解决的本质问题是什么(不是字面问题)

边界层:
  问题 ② 这是一次性需求 / 可复用工具?
  问题 ③ 范围是固定的 / 由用户输入?

立场层:
  问题 ④ skill 应该多大程度替用户做判断?
       事实陈述 / 概率指引 / 操作建议

形态层:
  问题 ⑤ 输出是简短摘要 / 完整答卷 / 交互式探索?

诚实层:
  问题 ⑥ 用户能接受的"承认不确定"的程度?
       藏在脚注 / 显眼标注 / 不确定时拒绝输出
```

每个 skill 的阶段 0 应该挑出真正模糊的 2-4 个问,作为理解卡里的"关键决策点"。

## §4 理解卡格式标准

见 `templates/requirement/intent-card.md`。

最小结构:

```
┌───────────────────────────────────────┐
│ 🧭 我对你需求的理解                      │
├───────────────────────────────────────┤
│ ① 你想解决的问题(一句话复述用户原话)     │
│ ② 你的真实意图(我推断的)                 │
│ ③ 关键决策点(2-4 个,带主张)              │
│ ④ 我打算给你的(一句话)                   │
│                                       │
│ [✓ 确认]  [✎ 修改某项]  [✗ 完全重来]    │
└───────────────────────────────────────┘
```

## §5 与 7 题画布的关系

理解卡 ≠ 7 题画布。两者关系是:

- **理解卡**:用户视角,2-4 个决策点,1 屏可读完,5 秒拍板
- **7 题画布**:工厂视角,WHO/WHAT/WHY/WHERE/WHEN/HOW/PARADIGM 完整骨架,后续阶段的对齐基准

执行顺序:**先理解卡(对人) → 再画布(对工厂)**。  
理解卡的几个决策点会变成画布相应字段的填写依据,**画布是理解卡的展开**,不是另起炉灶。

如果画布和理解卡对不上,以理解卡为准——因为理解卡是用户亲自确认过的。

## §6 进 demo-v0 的形态要求(C3=A 决策)

理解卡不只是一份 markdown 文档,**它必须在 demo-v0.html 里以"屏 -1"的形态实际渲染**——视觉化、可交互、可点击三个出口按钮。

理由:阶段 0 必须是产品的一部分而不是文档里的隐性流程,否则下次 skill 被调用时它根本不存在。

每个 skill 的 demo-v0 第一屏(屏 -1)就是这张理解卡。用户先过这一屏,再向下滚动看答卷主体。

## §7 失败处理

| 失败场景 | 处理 |
|---|---|
| 理解卡被用户"完全重来" | 跳出当前 skill,回到对话,问"那你想要的是什么?" 不要硬塞下一版理解卡 |
| 理解卡有 5 个以上"关键决策点" | 砍到 4 个以内,把次要的下放到画布或 demo 内的细调 |
| 用户对每个决策点都说"随便" | 给 2 个具体选项强制二选一,不接受"随便" |
| 决策点是设计偏好(色 / 字体 / 布局) | 不进理解卡,留到 S2/S3 自动决策 |
